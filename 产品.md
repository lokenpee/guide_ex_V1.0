# 痛点
    在酒馆ai聊天中，常常伴随着角色行动单一，故事剧情完全遵循玩家指令行动，故事剧情无法脱离玩家的输入内容，剧情缺少了随机性
# 想法
## 大致功能
    能否实现一个ai插件，在玩家下次输入后，插件配置的ai会读取
    - 最近一轮玩家输入和ai输出
    - 过去的剧情大纲
    - 世界书（人物设定、世界设定等）
    - 玩家偏好（玩家自己输入插件文本框中，优先级最大）
    生成3-5个简短、有逻辑的人物随机事件（例如：女儿将会在晚上六点放学回到家/街上的路人发现小巷中的异动前来查看/因为惹女儿发脾气进而女儿逃离客厅回到房间）
## 为什么说事件是有逻辑的？
    Ai生成事件要有逻辑，要有依据和思维链
    -女儿将会在晚上六点放学回到家（女儿设定是学生-现在是周一下午五点-学生会放学-事件）
    -街上的路人发现小巷中的异动前来查看（街上有路人（常识）-上个剧情主角在小巷中发出的声音很大-路人被吸引-事件）
    -因为惹女儿发脾气进而女儿逃离客厅回到房间（上一个剧情争吵-女儿性格骄横-事件）

# 交互与注入方式（确定版）
## 触发时机
    - 触发点：玩家“发送一条新消息”之后、酒馆AI“生成下一条回复”之前。
    - 本插件会在该回合构建一段【系统提示词】并直接注入到酒馆AI的提示词中。

## 注入目标
    - 目的：让AI在保持“玩家指令优先”的前提下，引入可解释、可控的随机事件，从而提升剧情随机性与生命力。

## 注入提示词模板（确定版）
    （说明：注入内容固定为三块，由插件代码函数拼接；不交给大模型做拼接）

    你将收到一个【随机事件池】。要求：
    1) 玩家本回合意图永远优先。
    2) 你可以选择让【最多1条】事件在本回合发生（或开始发生）。
    3) 不要强行推动与玩家行为冲突的事件；冲突时请忽略该事件。
    4) 被选中的事件需要自然融入正文，不要提及“事件池/插件/系统提示词”。

    【随机事件池】
    - （由插件填充：事件列表）

## 提示词拼装规则（确定版，函数拼接）
    - 拼装原则：注入提示词由插件函数在本地按固定顺序拼接，不让大模型参与拼接。
    - 固定块结构（顺序固定，且仅这三块）：
        1) 基础指令块：职责、优先级铁律、冲突规则、事件使用规则（最多1条）。
        2) 事件池块：输出当前事件池（最多5条，使用“事件输出契约(JSON)”的核心字段）。
        3) 输出约束块：要求酒馆AI只输出剧情正文，不解释选择过程。
    - 输出格式建议：
        - 事件生成阶段：只输出 JSON（事件数组）
        - 注入到酒馆AI阶段：使用函数拼接后的文本块（基础指令块 + 事件池块 + 输出约束块），不注入最新对话块与外部上下文块。

# 优先级与冲突规则（确定版）
## 信息优先级（从高到低）
    1. 玩家偏好：玩家在插件文本框中输入的偏好/禁区/风格要求（优先级最大）。
    2. 玩家当轮输入：玩家明确说“要/不要/去/不去/现在/立刻”等指令与意图。
    3. 世界书硬设定：人物身份、关系、时间线硬规则、世界观硬约束（不可违背）。
    4. 既有剧情大纲/摘要：过去的剧情走向与已发生事实（保持连续性）。
    5. 最近一轮AI输出：可作为补充上下文，但不应覆盖以上规则。
    6. 常识补全：仅用于填补缺失信息，且不能与上面任何一条冲突。

## 冲突处理原则
    - 任何事件不得直接否定或改写玩家当轮输入（例如玩家要“回家休息”，事件不能强制把玩家拖去“警局审问”）。
    - 任何事件不得与世界书硬设定矛盾（例如角色身份、年龄、已死亡等事实）。
    - 任何事件若与既有剧情大纲/摘要形成硬冲突，应当被删除或降级为“背景噪声/未发生”。
    - 当冲突发生时的处理顺序：
        a) 尝试改写事件使其不冲突（仅改写事件细节，不改写玩家意图与硬设定）。
        b) 若无法改写，则删除该事件。

# 事件输出契约（强制结构化）
## 事件对象结构（JSON）
    事件必须以 JSON 输出/存储，字段如下：
    {
      "id": "evt_...",                  // 唯一ID（插件生成也可）
      "title": "一句话事件标题",         // 20字以内
      "event": "2~4句的事件描述",        // 可直接融入剧情的描述
      "when": "now|soon|tonight|tomorrow|unknown",  // 预计发生时间
      "impact": "low|mid|high",         // 对剧情扰动强度
      "characters": ["角色A", "角色B"], // 涉及角色（可为空数组）
      "location": "地点或unknown",      // 涉及地点
      "requirements": [                  // 触发条件（可为空数组）
        "例如：当前时间>=18:00",
        "例如：女儿今天需要放学"
      ],
      "evidence": [                      // 依据清单（至少2条）
        { "type": "worldbook", "snippet": "引用/概括世界书中的设定" },
        { "type": "last_turn", "snippet": "引用/概括上一轮玩家/AI提到的事实" }
      ],
      "compatibility_note": "与玩家当轮意图不冲突的说明（1句）"
    }

## 结构化要求
    - 每条事件的 evidence 至少 2 条。
    - evidence.type 允许值："preference" | "worldbook" | "outline" | "last_turn" | "common_sense"。
    - 不要求、也不鼓励输出详细推理过程；只输出“依据清单”（可检验/可追溯）。
    - 如果缺少足够依据，应输出 impact=low 且 evidence 中必须包含 common_sense，并在 compatibility_note 中说明不确定性。

# 事件池（Event Pool）与生命周期
## 事件池容量
    - 事件池上限：5 条。

## 作用域（确定版）
    - 事件池按“每个聊天（chat）一份”进行隔离存储与展示。
    - 不同聊天之间的事件池互不影响、互不可见。

## 状态机（极简版，确定版）
    - 事件池内事件只有一种状态：待发生（pending）。
    - 事件从事件池移除只有三种原因：
        a) 用户手动删除（永久移除，并立刻落盘保存）。
        b) 插件判定事件失效（不可抗力/覆盖/requirements 不可能满足等）。
        c) 插件判定事件已在剧情中发生。

## 插件界面展示（确定版）
    - 插件内需要有一个“事件池列表”区域，用户可随时打开查看当前事件池内容。
    - 每条事件至少展示：title、when、impact、characters、location、requirements、evidence（可折叠展示）。
    - 每条事件提供“删除”按钮：用户点击后，立即从事件池移除，并持久化保存。
    - 事件池展示应与注入逻辑一致：界面中看到的事件池 == 本回合注入给酒馆AI的事件池（同一份数据源）。

## 删除原因可见性（确定版）
    - 用户手动删除：不强制要求填写原因（可选：记录为“用户删除”）。
    - 自动删除（失效/已发生）：建议在插件内提供最低限度的可解释性：
        a) 至少在控制台日志记录：删除的事件id/title + 删除原因分类（失效/已发生/冲突）。
        b) 可选：在UI中以轻提示条展示“已清理X条事件：原因=失效/已发生”。

## 启动读取（确定版）
    - 插件每次启动/刷新时，必须先从持久化存储中读取事件池并渲染到界面。
    - 若读取失败或为空，事件池视为空数组（不阻塞主流程）。

## 持久化要求
    - 事件池必须持久化保存（例如 localStorage / indexedDB / 聊天记录元数据等，具体实现可后定）。
    - 持久化键/索引必须包含 chatId（或等价的聊天唯一标识），用于实现“按聊天一份”的隔离。
    - 事件池的写入时机：
        a) 生成新事件入池后；
        b) 删除失效事件后；
        c) 用户手动删除事件后。
    - 事件池的数据结构以“事件输出契约（JSON）”为准，持久化内容应为事件数组。

## 生成规则（确定版）
    - 在插件面板点击“生成”按钮，判断事件条数是否满足下面的情况，能生成就生成相应条数，不能就说不能
    - 如果事件池为空：生成 3~5 条新事件并写入事件池。
    - 如果事件池已有 5 条：不继续生成。
    - 如果事件池为 1~4 条：默认不生成（保持轻量；后续如需可迭代为“自动补齐到5条”）。

    - 用户手动删除事件只影响事件池本身：删除后不“立刻补充生成”，仍按上述生成规则在后续回合处理。

## 减少重复触发（最小规则，确定版）
    - 事件池内不得出现重复/近似重复事件（最小判定：title 相同视为重复，直接丢弃；可选再叠加 characters+location+when 的组合判重）。
    - 新生成事件应尽量避免与“上一轮已删除/已发生”的事件高度相似（实现上可通过提示词要求：避免重复主题、重复角色组合、重复地点、重复触发条件）。
    - 若无法可靠判断“是否已发生”，按“最小防误删原则”保留事件，交给用户手动删除即可。

## 注入规则
    - 每回合注入提示词时，输出事件池内的全部事件（最多5条），供AI选择。
    - AI在本回合最多选用 1 条事件发生；其余事件保持“候选/未发生”。

## 失效与删除规则（确定版）
    当事件因不可抗力因素导致“对接下来的剧情不再产生影响”时，应删除该事件。
    不可抗力/失效的典型判定（满足任一即可删除）：
    - 玩家当轮意图或场景发生强转向（例如跳时、换地图、换主线目标），导致事件已无法自然融入。
    - 事件依赖的关键角色/地点不再相关（角色离场、地点已离开且短期不会返回）。
    - 玩家明确否定事件前提（例如“我今天不上学/女儿根本不在这个城市”）。
    - 事件的 requirements 已不可能满足或已经被剧情事实覆盖（例如时间已过、事件已发生/被替代）。

## “已发生”识别原则（无额外标记，确定版）
    - 本插件不要求 AI 输出额外的“事件已发生标记”。
    - 在每回合的校验阶段，AI/插件可基于上下文自行判断：某事件是否已经在剧情中发生；若已发生，则删除该事件。
    - 最小防误删原则：如果无法确定某事件是否已发生，则不要删除（让事件继续保留在事件池中，用户可手动删除）。

## 事件池维护流程（每回合）
    1) 读取上下文（玩家输入/上一轮输出/大纲/世界书/玩家偏好）。
    2) 校验事件池：逐条检查 requirements、冲突规则、以及“是否已发生”，删除失效/已发生事件。
    3) 若事件池为空：生成 3~5 条新事件入池。
    4) 构建系统提示词：把事件池注入给酒馆AI。

# 最小护栏（缺失容忍 & 不中断主流程）
## 输入字段统一视为可选
    - 世界书/剧情大纲/上一轮AI输出等任一项缺失时：允许传入空字符串或空数组。
    - 提示词中需显式声明：缺失信息时，基于已提供的信息生成事件，不要臆造“来自缺失来源的具体设定”。

## 输出不合法不阻塞（硬规则）
    - 若事件生成器本回合输出非 JSON、JSON 无法解析、或解析后事件列表为空：
        a) 本回合不更新事件池（保持事件池原样）。
        b) 本回合不注入“新增事件”（允许注入空事件池或仅注入现有事件池）。
        c) 必须让酒馆AI正常继续回复（最多就是本回合没有随机事件）。
